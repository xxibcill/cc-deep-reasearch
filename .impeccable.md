# Design Context

## Users

**Primary Users**: Developers and technical operators who use this tool in two distinct contexts:

1. **Ad-hoc researchers** - Developers running quick research queries from the CLI to inform their work. They need fast, reliable results with minimal friction and clear output formatting.

2. **Session operators** - Technical users monitoring long-running research sessions through the dashboard. They need deep visibility into agent behavior, LLM routing decisions, telemetry data, and execution traces to debug issues and understand research quality.

Both users are technical and comfortable with CLI tools, but they value interfaces that make complex multi-agent systems understandable through clear visualizations and well-organized data.

## Brand Personality

**Three-Word Personality**: Analytical, Transparent, Reliable

**Emotional Goals**:
- **Clarity first** - Complex multi-agent workflows should feel simple and understandable
- **Insight-driven** - Users should feel empowered to understand what's happening under the hood
- **Quiet confidence** - The interface should feel capable and trustworthy without being flashy

**Voice & Tone**:
- Technical but not academic
- Precise and information-dense, not sparse
- Professional but approachable
- Function over form, but form should still be thoughtful

## Aesthetic Direction

**Visual Tone**: Modern observability meets AI-native tooling

**References**:
- **Observability tools (Datadog, New Relic)** - For the metrics-rich, graph-heavy, real-time monitoring approach. The decision graphs, agent timelines, and telemetry visualization should feel at home in this domain.
- **Claude Code / Cursor IDE** - For the AI-native, minimal, workflow-focused aesthetic. Clean interfaces that prioritize visibility into agent behavior without unnecessary chrome.

**Anti-References**:
- **Static/legacy CLI tools** - Avoid feeling dated or purely terminal-bound. The dashboard should feel modern and alive.
- **Overly playful/whimsical** - This is serious research infrastructure, not a consumer app or toy.

**Theme**:
- **Dark mode primary** - The dashboard should feel like a modern developer tool with dark mode as the default experience
- **Data visualization focus** - Graphs, timelines, and decision trees are core to the experience, not afterthoughts
- **Subtle motion** - Real-time updates and live monitoring should feel responsive without being distracting

## Design Principles

1. **Make complexity visible, not overwhelming**
   - Multi-agent workflows are inherently complex. Show what's happening through clear visualizations (workflow graphs, decision trees, agent timelines) rather than hiding it.
   - Use progressive disclosure - show overview first, details on demand.

2. **Design for both quick queries and long-running sessions**
   - CLI output should be scannable and actionable for quick research.
   - Dashboard should support sustained monitoring with session persistence, historical comparison, and drill-down capabilities.

3. **Treat telemetry as a first-class feature**
   - Observability isn't an add-on - it's core to the value proposition.
   - Decision graphs, execution traces, and LLM route analytics should be prominent and well-designed.

4. **Leverage familiar patterns from developer tools**
   - Use conventions users already know from observability platforms, IDEs, and dashboards.
   - Status badges, timeline views, and metric cards should feel intuitive.

5. **Balance information density with breathing room**
   - Technical users want dense, information-rich interfaces.
   - But maintain clear visual hierarchy and avoid overwhelming users.
   - Use whitespace deliberately to separate concerns and guide attention.

## Technical Foundation

**Current Stack**:
- Next.js 14 + React 18 + TypeScript
- Tailwind CSS with custom design tokens
- shadcn/ui component patterns (cards, badges, buttons)
- D3.js for data visualizations
- Dark mode via next-themes
- IBM Plex Sans (body) + IBM Plex Mono (code)

**Design Tokens (from globals.css)**:
- Primary: HSL 212-214 (blue spectrum)
- Radius: 0.75rem (rounded corners)
- Typography: IBM Plex Sans for body, IBM Plex Mono for code/technical content
- Color roles: Background, foreground, card, muted, accent, destructive, ring, border

**Component Patterns**:
- Card-based layouts with subtle borders
- Semantic badges for status (success/warning/destructive)
- Responsive grid layouts
- Clean tabbed interfaces for switching views
