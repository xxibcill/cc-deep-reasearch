# Task 029: Rebuild PDF Export Around An HTML-First Pipeline

Status: Planned

## Objective

Refactor report export so HTML is a first-class intermediate artifact and PDF
generation becomes an HTML-to-PDF render step instead of a markdown-owned
operation.

## Problem Statement

The current implementation in
[`src/cc_deep_research/pdf_generator.py`](/Users/jjae/Documents/guthib/cc-deep-research/src/cc_deep_research/pdf_generator.py)
already converts markdown to HTML before calling WeasyPrint, but that HTML stage
is hidden inside `PDFGenerator.generate_pdf()`.

That creates several problems:

- the public API still models the feature as direct markdown-to-PDF conversion
- HTML cannot be saved, inspected, or reused as a standalone output
- markdown parsing, HTML document assembly, CSS selection, and PDF rendering are
  coupled in one module
- the CLI and docs are inconsistent:
  - [`src/cc_deep_research/reporting.py`](/Users/jjae/Documents/guthib/cc-deep-research/src/cc_deep_research/reporting.py)
    claims HTML support
  - [`src/cc_deep_research/config.py`](/Users/jjae/Documents/guthib/cc-deep-research/src/cc_deep_research/config.py)
    accepts `html`
  - [`src/cc_deep_research/cli.py`](/Users/jjae/Documents/guthib/cc-deep-research/src/cc_deep_research/cli.py)
    only exposes `markdown` and `json`

## Desired Architecture

The export pipeline should become:

1. research result -> canonical markdown report
2. markdown report -> HTML report document
3. HTML report document -> PDF file

Recommended implementation shape:

- keep markdown normalization in `MarkdownReportFormatter`
- introduce a dedicated HTML renderer responsible for:
  - markdown extensions
  - semantic section wrappers
  - HTML document shell
  - stylesheet selection
- narrow PDF code so it only renders already-built HTML to a `.pdf`
- route both `research --pdf` and `markdown-to-pdf` through the same HTML renderer

## Scope

- extract markdown-to-HTML behavior from `PDFGenerator`
- make HTML generation callable from application code and CLI code
- add a consistent HTML output path for research reports
- ensure PDF generation consumes HTML generated by the shared renderer
- preserve the existing report structure, wrapper classes, and current CSS intent
- update tests to validate both stage boundaries:
  - markdown -> HTML
  - HTML -> PDF

Out of scope:

- changing report content generation
- replacing WeasyPrint with another PDF backend
- redesigning report typography beyond what is needed to separate responsibilities
- adding browser-only rendering dependencies

## Acceptance Criteria

- the codebase exposes a shared HTML-rendering surface for report export
- PDF generation no longer parses markdown directly inside the final PDF-writing
  step
- `research` can emit HTML as a normal output format, or an equivalent explicit
  HTML export command exists
- `research --pdf` and `markdown-to-pdf` both use the shared HTML renderer
- HTML output preserves the existing section-aware wrapper classes used by the
  PDF stylesheet
- tests cover HTML generation without needing to inspect a binary PDF
- docs and CLI help no longer describe the feature as if markdown jumps straight
  to PDF

## Suggested Verification

- run `uv run pytest tests/test_markdown_to_pdf.py`
- add targeted tests for the HTML renderer and run them
- manually verify:
  - `cc-deep-research research --format html -o report.html "Topic"`
  - `cc-deep-research research --pdf -o report.md "Topic"`
  - `cc-deep-research markdown-to-pdf report.md`

## Rollout Plan

1. Introduce a small HTML renderer abstraction and move `_convert_to_html()`,
   wrapper logic, and stylesheet ownership into it.
2. Change `PDFGenerator` into a narrower HTML-to-PDF adapter that accepts HTML
   input plus CSS.
3. Update `ReportGenerator` and CLI flows so HTML is a real export target and PDF
   generation reuses that HTML output.
4. Refresh tests and docs after the new pipeline is stable.

## Notes For The Implementer

- Favor small, explicit objects over adding more branching to `PDFGenerator`.
- Keep one canonical source of CSS and section-wrapper behavior.
- Preserve current output compatibility where practical, but prefer a cleaner
  API boundary over preserving misleading method names.
- If backward compatibility is needed, keep a thin deprecated helper that calls
  the new staged pipeline rather than duplicating logic.
